# 💬 FreeChat 技术规范文档

> **版本**: v1.0 · **更新日期**: 2026-05-10

---

## 📖 概述

本规范定义 FreeChat 核心架构、数据协议及安全标准，确保系统可扩展性、安全性及低成本运行。

---

## 1. 🏗️ 核心架构

FreeChat 采用 **Cloudflare Serverless 全家桶** 架构，利用边缘计算实现低延迟通讯：

| 组件                                    | 用途                                  |
| --------------------------------------- | ------------------------------------- |
| **Runtime** — Cloudflare Workers (Hono) | 业务逻辑与 API 路由                   |
| **Database** — Cloudflare D1 (SQL)      | 用户信息、群组元数据、历史消息        |
| **Real-time** — Durable Objects (DO)    | 群组实时通讯与 WebSocket 连接状态管理 |
| **Storage** — Cloudflare R2             | 图片、语音、文件存储                  |
| **Cache/Meta** — Cloudflare KV          | 临时配置、高频访问的非敏感数据        |

---

## 2. 🔐 身份验证协议 (Auth)

系统采用 **双重 Hash + JWT** 机制，确保密码在传输过程及数据库泄露时均安全。

### 2.1 密码处理流程

```
┌──────────┐     SHA-256      ┌──────────┐    PBKDF2×100k     ┌──────────┐
│  前端层   │ ──────────────→  │  传输层   │ ────────────────→  │  后端层   │
│ 明文密码  │  生成 clientHash │ clientHash│  + salt 二次哈希  │ finalHash│
└──────────┘                  └──────────┘                    └──────────┘
```

| 阶段              | 操作                                                                         |
| ----------------- | ---------------------------------------------------------------------------- |
| **① 前端**        | 对原始密码进行 **SHA-256** 哈希，生成 `clientHash`（明文不离设备）           |
| **② 传输**        | 通过 HTTPS 发送 `clientHash`                                                 |
| **③ 后端 — 查询** | 从 D1 获取用户专属 `salt`（盐值）                                            |
| **④ 后端 — 哈希** | 使用 **PBKDF2（100,000 次迭代）** 对 `clientHash` 二次加盐，生成 `finalHash` |
| **⑤ 后端 — 签发** | 比对 `finalHash`，成功后签发含 `userId` 的 JWT                               |

## 3. 💬 聊天频道设计 (Messaging)

为区分私聊与群聊并优化 DO 使用成本，`channel_id` 遵循以下规范。

### 3.1 ID 格式与转义

用户 ID 可能包含特殊字符，故采用 **前缀区分 + URL 编码**：

| 类型        | 格式                        | 示例                     |
| ----------- | --------------------------- | ------------------------ |
| 🏠 **群聊** | `g:{group_uuid}`            | `g:8e2d...`              |
| 👤 **私聊** | `p:{encoded_a}:{encoded_b}` | `p:admin%3A01:user%4099` |

> **私聊生成逻辑**: `sort(userA_id, userB_id)` → `encodeURIComponent()` → 拼接

### 3.2 路由分片 (Sharding Strategy)

| 模式        | 策略     | 说明                                                   |
| ----------- | -------- | ------------------------------------------------------ |
| 🏠 **群聊** | 独立 DO  | 每个群组（`g:` 前缀）对应一个独立 DO 实例              |
| 👤 **私聊** | 哈希分片 | `hash(channel_id) % 100` 映射到 **100 个共享 DO** 实例 |

## 4. 🗄️ 数据库 Schema (D1)

### 4.1 消息表 `messages`

| 字段         | 类型        | 说明                                 |
| ------------ | ----------- | ------------------------------------ |
| `id`         | `TEXT (PK)` | 消息唯一 UUID                        |
| `channel_id` | `TEXT`      | 带 `g:` 或 `p:` 前缀，**无物理外键** |
| `userId`     | `TEXT`      | 发送者用户 ID                        |
| `content`    | `TEXT`      | 消息内容（支持 JSON 承载富媒体）     |
| `type`       | `TEXT`      | `text` · `image` · `file` · `system` |
| `createAt`   | `INTEGER`   | 毫秒级时间戳，用于排序               |

---

## 5. 🔌 实时通讯协议 (WebSocket)

| 阶段         | 说明                                                            |
| ------------ | --------------------------------------------------------------- |
| **连接建立** | 前端通过 `/ws/group/:groupId` 或 `/ws/private/:chatId` 发起连接 |
| **状态同步** | DO 内存维护 `sessions` 列表，一人发言、全员广播                 |
| **持久化**   | 消息广播同时，由 DO 异步写入 D1                                 |

---

## 6. 🛡️ 开发与安全规范

| 规范         | 要求                                              |
| ------------ | ------------------------------------------------- |
| **SQL 安全** | 严禁字符串拼接，必须使用 `db.prepare().bind()`    |
| **ID 转义**  | 读写 `channel_id` 时必须执行 `encodeURIComponent` |
| **错误屏蔽** | 禁止暴露 D1 / DO 原始错误堆栈，仅返回业务错误码   |
